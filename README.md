## About

I build **production generative AI systems** — not demos. My work sits at the intersection of applied LLM engineering, full stack product development, and operational reliability.

I ship **retrieval-augmented generation platforms** that run 24/7 for paying clients: hybrid retrieval with BM25 and dense vector search fused via reciprocal rank fusion, chunking strategies tuned per corpus, grounded generation with inline citation enforcement, and measured hallucination rates rather than marketing claims. Around the model layer I own the entire surface — **Next.js and TypeScript** on the front, **FastAPI and Python** on the back, **PostgreSQL with pgvector** underneath, streamed to the browser over Server-Sent Events.

Before AI development, I spent roughly **1.5 years as an L2 Support and Incident Management Analyst at HCLTech**, resolving ~40 requests daily at 98% SLA compliance across 24/7 rotational shifts using ServiceNow, Moogsoft, Autosys and Grafana. That background is not a detour — it is why I instrument, alert on, and degrade gracefully in every system I build. **Reliability is a feature, and I have carried a pager for it.**

I run **We Dev**, my own development studio, where client work forces the discipline that side projects never do: latency budgets, cost ceilings, security headers, secret scanning, and a product that has to work on Monday morning.

<br/>

A full-stack retrieval-augmented generation platform built and deployed for a real dental clinic through **We Dev**. It handles patient queries around the clock — treatments, procedures, pricing, appointment policy — grounded strictly in the clinic's own documentation, with every answer traceable to a source passage.

| Attribute | Detail |
| :--- | :--- |
| **Stack** | Next.js · TypeScript · FastAPI · Python · PostgreSQL + pgvector · LangChain · LLM APIs |
| **Scale** | 24/7 deployed production service · continuous live client traffic · multi-document knowledge corpus |
| **Performance** | Streamed token-by-token responses over SSE · hybrid BM25 + vector retrieval with Reciprocal Rank Fusion · tuned chunk sizing for domain-specific recall |
| **Security** | Server-side API key isolation · request rate limiting · sanitised inputs · secret scanning with gitleaks · hardened security headers |
| **Impact** | Replaced manual after-hours query handling with an always-available assistant; measured hallucination rate tracked against a curated evaluation set rather than asserted |
| **Repository** | [`sriram918offl/dental-clinic-ai-platform`](https://github.com/sriram918offl) |

This is my primary proof point as an AI engineer. It is not a notebook — it is a system with a client, an uptime expectation, and a cost ceiling. The engineering decisions that matter here are unglamorous: how documents are split, how retrieval failures degrade, how the model is prevented from answering when the corpus does not support an answer, and how latency is kept low enough that a patient does not close the tab. Working against a real corpus with real users forced a discipline around retrieval evaluation that no tutorial project produces.
